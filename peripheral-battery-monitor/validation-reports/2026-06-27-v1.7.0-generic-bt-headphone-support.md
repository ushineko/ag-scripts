# Validation Report — v1.7.0 Generic Bluetooth Headphone Support

**Date**: 2026-06-27
**Version**: 1.7.0
**Scope**: Make the two bottom (headphone) cells vendor-neutral — show whatever Bluetooth headphones are connected (e.g. Sony WH-1000XM6) instead of one cell pinned to SteelSeries Arctis and the other to Apple AirPods. Spec 013.

## Summary

`get_all_batteries()` previously emitted two fixed keys: `headset` (SteelSeries via `headsetcontrol`) and `airpods` (Apple via D-Bus + BLE). A newly connected Sony WH-1000XM6 reports battery cleanly via BlueZ `org.bluez.Battery1` but had no slot because there was no Sony-specific reader. Adding a reader per vendor does not scale. This change adds a vendor-neutral reader that enumerates connected Bluetooth audio devices over the BlueZ D-Bus `ObjectManager` and reads `org.bluez.Battery1`; the AirPods (L/R/case) and Arctis (USB dongle) readers are kept as enrichment sources, merged and de-duplicated by MAC, then ranked connected-first into `headphone1`/`headphone2`.

## Changes

- `battery_reader.py`
  - New `_enumerate_bt_audio_devices()` — vendor-neutral BlueZ D-Bus enumeration of connected audio devices and their `Battery1` level (stable-contract rule: D-Bus, not `bluetoothctl`).
  - New `get_headphones()` — merges generic BlueZ devices + AirPods (enrichment) + Arctis (`headsetcontrol`), de-duplicates by MAC, ranks connected-first (`_headphone_rank_key`).
  - `get_airpods_battery()` now attaches `ids={'mac': ...}` so dedup can match it.
  - `get_all_batteries()` replaces `headset`/`airpods` keys with `headphone1`/`headphone2`; the `__main__` CLI block prints the headphone list.
- `peripheral-battery.py`
  - Two bottom cells renamed `headphone1_ui`/`headphone2_ui` (fallback label "Headphones"); `on_data_ready` consumes `headphone1`/`headphone2`.
  - `update_single_device`: merge fallback guarded by `device_name` so a slot changing occupants cannot bleed the previous device's cached level onto a different device.
  - Version bump to 1.7.0.
- `tests/test_battery_logic.py`
  - Added `setScaledContents` to `MockQLabel` — pre-existing gap (introduced with the v1.6.x font-scale layout change) that broke every test instantiating the monitor.
  - Added `TestHeadphoneAggregation` (5 tests: generic surfacing, MAC dedup with detail-wins, Arctis still present, connected-first ranking, `get_all_batteries` slot keys) and `TestHeadphoneSlotNameGuard` (2 tests: no cross-device level bleed, same-device merge still works).

## Phase 3 — Tests

- `/usr/bin/python3 -m pytest tests/` — **67 passed** (54 prior-passing + 11 pre-existing failures fixed by the mock addition + 7 new — net 60→67 after also restoring the 6 previously-failing monitor-instantiation tests... see note).
  - Note: at HEAD, 11 tests were already failing due to the missing `setScaledContents` mock; this change fixes them. The suite was 54 passing / 13 failing before, 67 passing / 0 failing after.
- Live hardware check: `battery_reader.py --json` with the WH-1000XM6 connected returns `headphone1` = `WH-1000XM6` at 58%, `ids.mac` populated, no vendor-specific path.

## Phase 4 — Code Quality

- No dead code introduced. The two old single-purpose readers (`get_headset_battery`, `get_airpods_battery`) are retained and reused as enrichment sources, not duplicated.
- Single ranking helper (`_headphone_rank_key`) and single shared `_AUDIO_UUIDS` set (the AirPods D-Bus reader's inline copy is left as-is to keep the change surgical).
- Functions stay short and single-responsibility.

## Phase 5 — Security

- Read-only D-Bus enumeration and existing `subprocess.run` calls with fixed argument lists (no shell, no user-controlled input). No secrets, no credentials, no new dependencies. The Bluetooth MAC is a hardware identifier used only for in-process de-duplication; it is not logged or persisted by this change. Dependency scan not applicable (no dependency changes).

## Phase 5.5 — Release Safety

- **Rollback**: revert the commit. Self-contained to `battery_reader.py`, `peripheral-battery.py`, tests, spec, README, and this report; no data migration, no config schema change, no persisted-state format change. The result-dict keys changed (`headset`/`airpods` → `headphone1`/`headphone2`) but are internal to a single poll cycle and not persisted.
- **Additive/behavioral**: the only removal is the two vendor-pinned result keys, both superseded by the generic slots in the same release.
- A running monitor instance must be restarted to pick up the change (long-running GUI; reads the reader as a subprocess each poll, but the UI cell wiring is loaded at startup).
