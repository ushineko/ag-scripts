# Validation Report — v1.10.0 Live AirPods Battery via AAP/L2CAP

**Date**: 2026-07-17
**Version**: 1.10.0
**Scope**: Read live AirPods Left/Right/Case battery over Apple's Accessory Protocol (AAP) on an L2CAP channel, so the Headphone slot shows a real percentage.

## Summary

BlueZ never exposes `org.bluez.Battery1` for AirPods on this stack (PipeWire owns
the HFP profile, so the Apple HFP battery report never reaches BlueZ — confirmed in
v1.9.2's investigation, including with experimental mode on). The prior BLE
advertisement scan only returned data when the AirPods happened to be broadcasting
battery, so the level was usually absent (and, when present, a frozen last-known
value). This change reads battery **directly over AAP/L2CAP** (PSM 0x1001) — the
mechanism LibrePods uses — which is reliable and gives live L/R/Case levels. A
standalone proof-of-concept read a live value first; this integrates that into the
reader.

## Changes (`battery_reader.py`)

- AAP protocol constants + `_parse_aap_battery` (packet → `{component: (level,
  charging)}`, with spacer/terminator validation and disconnected-record drop) and
  `_aap_pods_to_info` (→ `BatteryInfo`; level = min of ear pods, charging if any
  pod charging, `details` = L/R/Case for the existing UI).
- `_read_airpods_aap(mac)`: opens L2CAP PSM 0x1001 (BR/EDR), runs the ACK-gated
  handshake → set-features → request-notifications sequence (with time-based
  fallbacks and one retry), parses the first battery packet, and always closes the
  socket. Bounded to ~8s.
- `get_airpods_battery`: now prefers the cached AAP read; falls back to the BLE
  scan, then presence-only. The 120s disk cache (added in v1.9.2) is reused and
  renamed generically (`_read/_write_airpods_cache`, `~/.cache/.../airpods_battery.json`).
- Added `import socket`, `import select`. Removed a dead `return None`.

## Phase 3 — Tests

- `QT_QPA_PLATFORM=offscreen /usr/bin/python3 -m pytest tests/` — **92 passed**
  (83 prior + 9 new), stable across two runs. New `tests/test_airpods_aap.py`
  covers the parser (AAP-docs worked example, disconnected-record drop,
  spacer/terminator rejection, non-battery packets, out-of-range level) and the
  level/status derivation.
- **Live (AirPods Pro connected)**:
  - `battery_reader.py --json` over 4 runs: 2.4–3.5s each, all returned
    `headphone1` level 90, `details {left:91, right:90}`, status Discharging.
    First run (cache cleared) did the AAP read; the rest hit the 120s cache.
  - Standalone AAP probe earlier returned L/R = 92% independently, confirming the
    protocol.
  - Real app (v1.10.0): the slot shows "Papa's AirPods Pro — L:90% R:90% —
    Discharging" (screenshot), live rather than the previous frozen value.
- The AAP read is an integration boundary (Bluetooth hardware); verified against
  the real device. The parser is unit-tested.

## Phase 4 — Code Quality

- The BLE scan and its bound are retained only as a fallback; no duplicated logic.
  The AAP read is bounded and always closes its socket (mirrors the v1.9.2 hang
  fix). Parser is small and validated.

## Phase 5 — Security

- No new dependencies (`socket`, `select` are stdlib). The socket connects only to
  the user's own paired AirPods (MAC obtained from BlueZ D-Bus enumeration,
  filtered by device name); all sent packets are fixed protocol constants; received
  packets are parsed with explicit length/field checks. No user-controlled input
  reaches the socket. The cache holds only a MAC, timestamp, and battery fields —
  no secrets. Best-effort file I/O with failures ignored.

## Phase 5.5 — Release Safety

- **Rollback**: revert the commit. The cache file is disposable (regenerated next
  poll); no schema/config change. The old `airpods_ble.json` cache path is simply
  left unused.
- **Additive/behavioral**: AAP is tried first; if it fails, behavior degrades to
  the previous BLE-scan-then-presence path. Non-AirPods paths are untouched.
