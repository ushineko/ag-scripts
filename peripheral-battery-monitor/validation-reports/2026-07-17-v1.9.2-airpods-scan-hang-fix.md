# Validation Report — v1.9.2 AirPods BLE-Scan Hang / Slot-Switch Fix

**Date**: 2026-07-17
**Version**: 1.9.2
**Scope**: Fix the Headphone slot not switching to AirPods when connected.

## Summary

With v1.9.1's 15s refresh, connecting AirPods left the Headphone slot stuck on
"Disconnected". Root cause (diagnosed live with AirPods Pro connected): the AirPods
L/R/case **BLE advertisement scan** in `get_airpods_battery()` takes ~5s and
**intermittently hangs** (observed >45s; `scanner.stop()` wedging on a busy BlueZ
adapter). It ran on every 15s poll, so a hang exceeded the reader's 25s subprocess
timeout → the worker got no data → the slot (no offline cache) fell back to
Disconnected; a hung scan could also degrade the adapter and cascade. The data
layer was otherwise correct — `_find_airpods_via_dbus()` resolves AirPods presence
quickly.

## Changes (`battery_reader.py`)

- `_ble_scan_for_airpods`: `scanner.stop()` is now bounded (`asyncio.wait_for(..., 3s)`
  in a `finally`), and the scan sleep trimmed 5s→4s, so a stuck stop cannot wedge
  the reader.
- `get_airpods_battery`: the whole BLE scan is hard-capped
  (`asyncio.wait_for(..., 9s)`), and its result is cached on disk
  (`~/.cache/peripheral-battery-monitor/airpods_ble.json`, TTL 120s) — the reader is
  a fresh process per poll, so an in-memory cache would not survive. The fast
  BlueZ presence/`Battery1` path is unchanged and still runs first, so the slot
  switches to the AirPods immediately regardless of the scan.
- `import time` added.

## Phase 3 — Tests

- `QT_QPA_PLATFORM=offscreen /usr/bin/python3 -m pytest tests/` — **83 passed**.
- **Live (AirPods Pro connected)**:
  - Reader over 5 runs: run 1 = 6.0s (scan + cache), runs 2–5 = 2.0–2.6s (cached);
    all resolved `headphone1 = Papa's AirPods Pro`. **No hangs.**
  - Forced fresh scan ×4 (cache cleared each time): 5.6 / 11.0 / 10.6 / 5.7 s —
    **max 11.0s**, well under the 25s worker timeout. Before the fix, runs hit
    >45s.
  - Real app (v1.9.2): the Headphone slot switched from "Disconnected" to
    "Papa's AirPods Pro — Connected" (screenshot confirmed).

## Phase 4 — Code Quality

- Change is contained to the AirPods reader. The cache helpers are small and
  guarded (all I/O in `try/except`); no behavior change to non-AirPods paths.

## Phase 5 — Security

- No new dependencies (bleak/asyncio already used). The cache is a local JSON file
  in the user's cache dir containing only a MAC, timestamp, and battery fields —
  no secrets. Best-effort file I/O, failures ignored.

## Phase 5.5 — Release Safety

- **Rollback**: revert the commit. The cache file is disposable (safe to delete;
  regenerated on next poll). No schema/config change.
- **Additive/behavioral**: bounds and caches an existing scan; the fast presence
  path is unchanged.

## Out of Scope (documented, not fixed here)

- **AirPods battery level** remains unavailable on this setup. AirPods report
  battery over Apple's HFP extension, but PipeWire owns the HFP profile natively so
  BlueZ never sees it and cannot expose `org.bluez.Battery1` — confirmed live:
  enabling BlueZ `Experimental` and forcing the HFP profile still produced no
  `Battery1`. Reliable AirPods battery needs ofono as the HFP backend or a
  dedicated AAP daemon (e.g. LibrePods). Recorded in the README AirPods note.
