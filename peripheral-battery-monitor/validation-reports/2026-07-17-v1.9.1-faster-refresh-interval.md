# Validation Report — v1.9.1 Faster Device-Refresh Interval

**Date**: 2026-07-17
**Version**: 1.9.1
**Scope**: Reduce the full device-refresh interval so the configurable slots pick up device connect/disconnect promptly.

## Summary

With the v1.9.0 configurable slots, connecting a headphone (e.g. plugging in the
Arctis) did not switch the slot for up to 10 minutes because the full refresh timer
was set to 600000 ms. The data layer was already correct (`get_headphones()` ranks
the newly connected device as `headphone1`); only the poll cadence was too slow for
the now-dynamic slots. Interval reduced to 15 s.

## Changes

- `peripheral-battery.py` — `setup_timer`: `self.timer.start(600000)` → `15000`.
  Version → 1.9.1.
- `README.md` — version, Changelog v1.9.1.

## Analysis

- A full poll (`battery_reader.py --json`) takes ~1.5 s and runs in a worker thread,
  so it does not block the UI. `update_status()` skips a tick if a worker is still
  running, so overlapping polls cannot pile up.
- The only expensive path, the AirPods BLE scan, runs **only** when AirPods are
  connected without a D-Bus `Battery1` level (`get_airpods_battery`: the scan is
  gated on `is_connected`). Arctis (USB/headsetcontrol) and Bluetooth headsets with
  `Battery1` never trigger a scan, so 15 s polling is cheap in the common case.
