**Date**: 2026-04-27

### Scope

The v3.0.0â€“v3.1.4 KWin rule (`Pinball FX Gamescope Placement` in `~/.config/kwinrulesrc`) matched only on `wmclass=gamescope`. KWin force-applies `position`/`size`/`fullscreen`/`noborder` to *every* matching window, so any concurrent gamescope session â€” Battle.net launcher, a Steam game launched through gamescope, even `gamescope -- alacritty` for debugging â€” was force-pinned to the portrait monitor at portrait geometry. The rule also persisted in `kwinrulesrc` between Pinball FX runs, so the hijack outlived the wrapper.

Two changes:
1. **Title-substring match.** Rule now requires `title=PinballFX, titlematch=2` in addition to `wmclass=gamescope`. Gamescope propagates the focused inner window's title to its outer xdg-toplevel, so UE-Pinball's `"PinballFX"` becomes the gamescope window's caption. Other gamescope sessions (`"Alacritty"`, etc.) no longer match.
2. **Transient rule install.** Wrapper's `cleanup()` trap now calls `install_kwin_rule.py --uninstall` on `EXIT INT TERM`. The rule no longer survives the wrapper.

### Empirical verification

Both changes were validated live, before being coded, against the user's running system (Plasma 6, gamescope 3.16.21, KWin Wayland) with two concurrent gamescope sessions: `pinball_fx_wrapper.sh` for Pinball FX and `gamescope -- alacritty` as a control.

**Window-property dump via KWin scripting** (`/tmp/kwin-window-dump.js` loaded through `org.kde.kwin.Scripting.loadScript`):

```
GAMESCOPE | caption="PinballFX" | resourceClass="gamescope" | resourceName="gamescope" | pid=3836522 | fullScreen=true | geom=2560,0,1440x2560 | output=DP-2
GAMESCOPE | caption="Alacritty" | resourceClass="gamescope" | resourceName="gamescope" | pid=3861508 | fullScreen=true | geom=2560,0,1440x2560 | output=DP-2
```

Confirmed: both gamescope windows had `resourceClass="gamescope"`, both were force-fullscreened at the portrait monitor's geometry â€” the rule was hijacking the alacritty session live. Captions differed (`"PinballFX"` vs `"Alacritty"`), establishing that title-match could discriminate.

**Dynamic-match test** â€” temporarily patched the rule to `title=Alacritty, titlematch=2`, reconfigured KWin, then ran a script that called `w.fullScreen = false` on both gamescope windows and re-checked 200 ms later:

```
[PinballFX] before: fullScreen=true â†’ [PinballFX] 200ms later: fullScreen=false   (rule released; caption no longer matched)
[Alacritty] before: fullScreen=true â†’ [Alacritty] 200ms later: fullScreen=true    (Force re-asserted; caption matched)
```

Confirmed: KWin re-evaluates rule matches dynamically when caption changes. The title-substring approach therefore behaves correctly at runtime, not just at window creation. Original `kwinrulesrc` was restored before any code changes.

### Out of Scope / Known Tradeoffs
- **Startup race**: between gamescope's outer surface mapping and UE setting the inner window's title, the gamescope caption is unset. KWin's rule won't match during that brief window, so gamescope may briefly appear at default placement before snapping to the portrait monitor once the title is set. UE-Pinball sets its title early (engine-init time), so the flash is sub-second; if it ever becomes objectionable, a KWin scripting + PID-match approach would eliminate it (the wrapper already captures `GS_PID=$!`).
- **Wrapper-crash leak**: if the wrapper is `SIGKILL`ed, the `EXIT` trap won't fire and the rule is left in `kwinrulesrc`. The next wrapper invocation will overwrite the geometry but won't actively delete the orphan; user can run `install_kwin_rule.py --uninstall` manually or just relaunch (and re-exit cleanly). Leak is benign because the rule still title-matches `PinballFX` only.
- **Concurrent inner caption collision**: a future game whose inner window title coincidentally contains `"PinballFX"` would match. Astronomically unlikely.
