### Phase 5: Security Review
- Dependencies: No third-party Python packages declared (only the system-installed PyQt6 and stdlib `urllib`/`json`/`tomllib`/`subprocess`/`signal`/`pathlib`/`dataclasses`/`logging`). pip-audit not run because there is nothing to scan.
- Token handling:
  - Token is read from `~/.config/slack-presence-toggle/token` (mode 600).
  - `install.sh` sets `chmod 700` on `~/.config/slack-presence-toggle/`.
  - Token never appears in any log statement; only the path appears in `log.warning` (`token file %s not found`).
  - HTTPS-only API endpoints, `Authorization: Bearer …` header.
- subprocess: `notify-send` invoked via `subprocess.Popen` with list-form arguments (no `shell=True`). `os.system`/`eval`: not used.
- Hardcoded secrets: none in source.
- OWASP Top 10:
  - A01 Broken access control: N/A (single-user desktop app).
  - A02 Cryptographic failures: N/A (token-as-bearer over HTTPS, file mode 600).
  - A03 Injection: N/A (no SQL, all subprocess calls list-form, all HTTP body construction via `urllib.parse.urlencode` or `json.dumps`).
  - A04 Insecure design: state machine pure-logic, dependencies injected for testability.
  - A05 Security misconfig: install.sh enforces directory mode 700, token mode 600.
  - A06 Vulnerable components: only PyQt6 (system package) and stdlib.
  - A07 Auth failures: surfaced via API health state machine (`invalid_auth`, `token_revoked`, `missing_scope`).
  - A08 Software integrity: N/A (no auto-update mechanism).
  - A09 Logging failures: token never logged.
  - A10 SSRF: N/A (no URL construction from user input).
- Anti-patterns: None found.
- Status: PASSED
