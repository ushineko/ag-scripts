## Validation Report: QProcess-Based Async Scanner (v1.7)
**Date**: 2026-04-19
**Spec**: 008-qprocess-async-scanner
**Status**: PASSED

### Phase 3: Tests
- Test suite: `/usr/bin/python3 -m pytest tests/ -q`
- Results: 88 passing, 0 failing (same count as v1.6 — the auto-refresh-related tests now work against the simplified MainWindow that delegates in-flight tracking to the scanner)
- Live smoke: a standalone `start_async_scan()` call against the user's real KWin + journalctl returned all 6 currently-open VSCode captions. End-to-end validated.
- Status: PASSED

### Phase 4: Code Quality
- Deleted code:
  - `_ScanWorker` class (10+ lines)
  - `_scan_thread`, `_scan_worker` attributes
  - `_on_scan_thread_done` method
  - `QThread` / `QObject` imports in `vscode_launcher.py`
  - Thread quit/wait logic in `closeEvent`
- `_trigger_background_scan` shrank from ~25 lines of thread plumbing to 5 lines (visibility check + `start_async_scan()`)
- Duplication: two scan paths exist (sync + async). Shared helpers (`_build_action_script`, `parse_captions_from_journal`, `caption_matches_label`, `running_labels`) are reused. The remaining duplication is the subprocess orchestration — fundamentally different between blocking and event-driven, not worth forcing into a shared abstraction
- Encapsulation: in-flight state lives on the scanner, not scattered across MainWindow
- Status: PASSED

### Phase 5: Security Review
- No new subprocess surface — same `qdbus6` and `journalctl` invocations as before
- Argument vectors unchanged (still list-form, no shell interpretation)
- Temp-file creation still via `NamedTemporaryFile(delete=False)` with unique suffix; unlinked in `_finish_async`
- JSON parsing still via `json.loads`; no pickle, no eval
- One new consideration: `QProcess.errorOccurred` connected alongside `finished` so a "failed to start" failure (e.g., `qdbus6` disappears mid-session) doesn't hang the state machine waiting for a `finished` that never comes
- Status: PASSED

### Phase 5.5: Release Safety
- Change type: refactor (same behavior, different machinery)
- Rollback plan: `git revert HEAD` restores v1.6 thread-based implementation. Both the thread-based and QProcess-based versions produce the same `scan_finished(list | None)` payload, so callers are unaffected by the choice.
- Backward compatibility: full. Config schema unchanged. Desktop entry unchanged. Install / uninstall unchanged. Zsh hook unchanged.
- Resource use: now creates 3 QProcess per scan cycle (load, run, log) plus 1 fire-and-forget unload QProcess. All `deleteLater`'d in `_finish_async`. Peak memory per cycle is ~4 small QProcess wrappers, deleted once the event loop next returns. No leak under repeated polling (verified by eye against `_finish_async`'s reset block).
- Status: PASSED

### Phase 6.5: Spec Reconciliation
- Spec 008 created; all 12 acceptance criteria checked; status COMPLETE

The implementation-notes section of spec 007 is historically correct for v1.6; spec 008 supersedes its *thread* details for v1.7+.
- Status: PASSED
