## Validation Report: v3.5.5 macOS popup-focus + window-activation fixes
**Date**: 2026-06-21
**Commit**: pre-commit
**Status**: PASSED

### Problems (reported from the running macOS app)
1. Invoking the popup hotkey also brought the launcher's own main window to the
   front.
2. Activating a running workspace (popup commit / row "Activate") did nothing —
   the existing VSCode window wasn't raised.

### Root causes & fixes
1. The macOS "show main window on app activation" handler (v3.5.1, for Spotlight
   relaunch) fired when the popup grabbed focus and made the app active. Added a
   suppression window: `_on_hotkey_pressed` sets `_suppress_activation_until =
   monotonic() + 2s`, and the activation handler now also bails if that deadline
   hasn't passed or the popup is visible.
2. Activation went through `WindowScanner.perform_window_action(ACTION_ACTIVATE)`,
   which uses KWin scripting (KDE-only) — a no-op on macOS. Added
   `Launcher.focus_workspace()` (runs `code <path>` without `--new-window`) and a
   `MainWindow._activate_workspace_window()` that branches: macOS → `code`,
   Linux → KWin. Both `_on_activate_requested` and `_launch_or_activate` use it.

### Phase 3: Tests

- Updated the activate-handler test to force the Linux branch (it asserts the
