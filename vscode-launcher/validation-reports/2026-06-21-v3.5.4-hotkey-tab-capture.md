## Validation Report: v3.5.4 Settings hotkey accepts Tab combos
**Date**: 2026-06-21
**Commit**: pre-commit
**Status**: PASSED

### Problem
The macOS Carbon hotkey backend accepts Tab-based combos (e.g. Ctrl+Shift+Tab),
but the Settings field is a `QKeySequenceEdit`, which consumes Tab/Backtab for
focus traversal — so a Tab combo could never be entered from the UI.

### Fix
`HotkeyEdit(QKeySequenceEdit)` overrides `event()`: a Tab/Backtab KeyPress is
routed to `keyPressEvent` (the sequence-recording path) and consumed, instead
of falling through to focus navigation. Used in `SettingsDialog`. The hint text
now mentions Tab is allowed.

### Runtime verification (macOS 26.5, M1 Max)
- `HotkeyEdit` records `Ctrl+Shift+Tab` from a synthesized key event.
- Backend registration: `set_binding` returns True for Ctrl+Shift+Tab,
  Ctrl+Tab, Meta+Shift+Tab, Alt+Tab (Carbon accepts them).
- Full app (frozen `.app`, reinstalled): with `global_hotkey` set to
  `Ctrl+Shift+Tab`, the daemon resolves the Carbon backend and binds it
  (`tray_setup_ok == True`).

### Limitations
- As with v3.5.3, the live keypress (Ctrl+Shift+Tab → popup) is a manual check.
- A global Tab combo overrides that key everywhere while the launcher runs
  (VSCode/browser tab cycling included) — documented in the changelog; the user
  can rebind in Settings.
