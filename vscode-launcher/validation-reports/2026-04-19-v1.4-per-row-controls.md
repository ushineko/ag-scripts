### Phase 4: Code Quality
- Dead code: none
- Duplication: none — action-script plumbing shares the same load/run/unload pattern as enumeration, but the invocation data differs (label + action vs. fixed enumeration). A shared "run-script-and-read-journal" helper would save ~20 lines at the cost of clarity; kept separate for readability
- Encapsulation: `_build_action_script` is a pure function (testable without any process calls); the action-dispatch branch is resolved at script-generation time rather than inside the JS, so the generated JS has no dead `if` branches for the un-picked action
- Defensive: `json.dumps(label)` for JS-literal safety, `int(stdout)` + range-check for the script id, `try/finally` for unload cleanup — all pre-existing patterns extended to the new call path
- Refactorings: none needed
- Status: PASSED

### Phase 5: Security Review
- New surface: the action script embeds a `label` parameter into KWin JS source
  - Labels flow from VSCode's recents (already trusted to name files / folders on disk) or from user-edited mapping keys
  - Escaping via `json.dumps` ensures any character — including `"`, `\`, newline, unicode — produces a valid JS string literal that cannot break out into executable context
  - Covered by `test_label_with_quotes_is_escaped` which verifies the escape is actually applied
- KWin API usage:
  - `w.closeWindow()` is VSCode-friendly: it triggers the normal close flow, respecting unsaved-changes prompts. We never force-kill
  - `workspace.activeWindow = w` is a standard property assignment; no side effect beyond raise + focus
- All subprocess invocations remain list-form; no `shell=True`; no `eval` / `exec`
- Fresh script files created with `tempfile.NamedTemporaryFile(delete=False, ...)`; unlinked in `finally`; unique per-call names via `tempfile` (no TOCTOU, no path collision)
- Script unload in `finally` prevents KWin registry leaks even if a failure occurs mid-run
- Status: PASSED
