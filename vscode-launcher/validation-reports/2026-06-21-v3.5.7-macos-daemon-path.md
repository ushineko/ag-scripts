## Validation Report: v3.5.7 macOS daemon PATH / `code` resolution
**Date**: 2026-06-21
**Commit**: pre-commit
**Status**: PASSED

### Problem (root cause behind several macOS symptoms)
Activating a running workspace did nothing, and launching workspaces from the
menu-bar daemon silently failed. The LaunchAgent inherits launchd's minimal
PATH (`/usr/bin:/bin:/usr/sbin:/sbin`), which excludes `/usr/local/bin` where
`code` is symlinked. Confirmed live: under that PATH `shutil.which("code")`
returns None (but `which("open")` works). So every `code`-based action from the
daemon bailed. Running-state detection kept working only because it uses IPC,
not `code` — which is why earlier symptoms looked partial.

### Fix
1. `resolve_code()`: returns an absolute path to `code` — `$PATH` first, then
   `/usr/local/bin/code`, `/opt/homebrew/bin/code`, and the VSCode `.app`'s
   bundled `bin/code`. `launch_workspace`, `focus_workspace`, and
   `_vscode_app_path` all use it.
2. The generated LaunchAgent now sets `EnvironmentVariables.PATH` to include
   `/usr/local/bin` and `/opt/homebrew/bin` (defense in depth for any
   subprocess).
